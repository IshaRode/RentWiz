# RentWiz Architecture & Workflow Overview

RentWiz is an end-to-end AI-powered real estate market intelligence platform. It identifies underpriced rental properties in major Indian cities by comparing real, live market listings against a machine learning model's prediction of "Fair Market Rent."

Here is a detailed, step-by-step breakdown of how the entire system works from the ground up.

---

## 1. The User Workflow (What the User Sees)
1. **Landing & Dashboard**: The user opens the RentWiz web app. They see a sleek, modern dashboard displaying top rental deals in their selected city.
2. **Filtering**: The user filters for "Bangalore, 2 BHK, Good Deals".
3. **Card Data**: Each property is displayed on a `DealCard` that highlights:
   - The **Actual Rent** requested by the landlord.
   - The **Predicted Fair Rent** calculated by the ML model.
   - The **Savings** (e.g., "+₹33,400/mo") and a colored badge (`Good Deal`, `Fair`, `Overpriced`).
   - A short, human-readable sentence explaining *why* it's a good deal (generated by AI).
4. **Action**: The user clicks the **"View Listing →"** button, which securely redirects them to the actual MagicBricks property page so they can contact the broker/owner.

---

## 2. Data Acquisition (Web Scraping)
Before the user ever logs on, the system needs live data.

**How it works:**
- We built a custom **Selenium-based headless web scraper** (`magicbricks_scraper.py`).
- The script automatically navigates to MagicBricks.com, filtering for specific cities (e.g., `Mumbai`, `New-Delhi`, `Bangalore`) and property sizes (e.g., `2-BHK`, `3-BHK`).
- **Data Extraction**: It scrolls the page to load dynamic elements and extracts:
  - `Title`, `Location`, `Area (sqft)`, `Bathrooms`
  - `Actual Rent` (passing through a robust regex parser to filter out bad data like "per sqft" rates or sale prices like "1.2 Cr").
  - `Listing URL` (The actual `magicbricks.com/...` link).
- **Storage**: Once cleaned, this data is bulk-inserted into a **Supabase (PostgreSQL)** database into the `scraped_listings` table, tagged with `source = 'scraped'`.

---

## 3. The Machine Learning Engine (The "Brain")
To determine if a property is a "deal", we need to know what it *should* cost.

**The Model:**
- We use a **Gradient Boosting Regressor (`GradientBoostingRegressor`)** built with Scikit-Learn.
- **Training**: The model was trained offline (`ml/train.py`) on a massive dataset of historical Indian rental properties. It learned the complex, non-linear relationships between a property's features (`City`, `BHK`, `Size in sqft`, `Furnishing Status`, `Bathrooms`) and its rental price.
- **The Pipeline**: The model is bundled with an automated preprocessor (`OneHotEncoder` for text like City/Furnishing, `StandardScaler` for numbers) into a single artifact: `full_pipeline.joblib`.

**Inference (Live Prediction):**
- When the API loads, it loads `full_pipeline.joblib` into memory.
- For every scraped property fetched from the database, the backend passes the property's specs through the model.
- The model outputs an integer: the **Predicted Fair Rent**.
- **Deal Scoring Formula**: `Predicted Rent - Actual Rent`. 
  - If a property's actual rent is ₹45,000 but the model says it should cost ₹78,400, the score is `+₹33,400`. The system classifies it as a **`good_deal`**.

---

## 4. The Generative AI Integration (The "Assistant")
Raw numbers and percentages can be dry. We use a Large Language Model (LLM) to act as a real estate agent explaining the deal to the user.

**The AI Model:**
- We use **Google's Gemini AI** API (`gemini-1.5-flash` or similar, configured via the `google-generativeai` SDK in `ai_explainer.py`).
- **The Prompt**: The backend constructs a prompt feeding the LLM the property details:
  > *"Analyze this rental: 3BHK in J P Nagar, Bangalore. Actual rent: ₹45,000. Predicted fair market rent: ₹78,400. Write a one-sentence, punchy explanation of why this is a good deal."*
- **The Output**: Gemini returns a human-friendly string (e.g., *"At just ₹45k, this spacious J P Nagar 3BHK is massively underpriced compared to its ₹78k fair market value, making it an absolute steal."*). This string is attached to the API response.

---

## 5. The Backend (FastAPI)
The central nervous system connecting the Database, the ML Model, the AI, and the Frontend.

- Built with **Python FastAPI**.
- When the frontend requests `GET /api/v1/best-deals`:
  1. The API queries **Supabase**, prioritizing rows where `source='scraped'` so users see real data.
  2. It iterates through the listings, running them through the loaded **Scikit-Learn pipeline** to get the prediction and compute the `deal_score`.
  3. It asynchronously requests **Gemini** to generate the explanations.
  4. It returns a strictly typed JSON array (validated by **Pydantic**) containing all the data, scores, and URLs back to the frontend.

---

## 6. The Frontend (Next.js)
The presentation layer that creates the premium user experience.

- Built with **React** and **Next.js (App Router)**.
- Styled using **Tailwind CSS** and heavily customized with **Aceternity UI** and **Framer Motion**.
- Components like `<DealCard />` use conditional logic to apply glowing borders (Emerald for good deals, Red for bad deals).
- The UI handles the API state, displaying loading skeletons while FastAPI computes predictions, and ultimately rendering the highly visual, data-rich cards with working external links to MagicBricks.

---
### Summary
**Web Scraper (Selenium)** → grabs raw listings → **Supabase** → stores data → **FastAPI** → queries data → runs through **Sklearn ML Model** to score the deal → asks **Gemini** to explain the deal → returns JSON to **Next.js** → User clicks to rent!
